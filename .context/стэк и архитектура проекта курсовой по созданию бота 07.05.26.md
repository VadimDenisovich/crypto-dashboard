
# Стэк:

React + python (postgresql + redis + ccxt + ccxt.pro + ta) 

Binance + bybit testnet 

Использовать иквидные и предсказуемые пары: 
BTC/USDT + ETH/USDT

# Архитектура всего проекта 

5 независимых сервисов:  
1. Frontend 
2. Backend API 
3. Trading Bot Engine — стратегии, ордера и все решения
4. Backtest Engine
5. Data&State Layer — PostgerSQL + Redis 

Связь через REST API и WebSocket. 

## Архитектура Frontend 

посмотреть биржи + сделать фигма проект для начала 

## Архитектура Backend 

**REST Endpoints**:    
- `GET/POST /api/strategies` – список и создание/обновление конфигураций.
- `GET /api/account` – баланс с биржи (проксируется из движка или из кэша).
- `GET /api/trades` – история сделок.
- `POST /api/backtest` – запуск бэктеста с переданными параметрами, возвращает id задачи.

**WebSocket `/ws/updates`**:
- Принимает подписку от UI на события: `new_trade`, `balance_update`, `engine_status`.
- Получает эти события от Trading Engine через Redis Pub/Sub и пересылает клиенту.

Это всё примерно, надо будет детальнее обсудить архитектуру после составления макета и функционала. 

## Архитектура Trading Bot Engine 

clean архитектура, чтобы доменная логика не зависела от бирж и баз данных.

**Слои внутри Engine:**

**Domain Layer** (ядро):
- `Strategy` (абстрактный класс): метод `on_candle(candle) -> Signal | None`.
- `Signal` – buy/sell с параметрами.
- `Position`, `Order`, `Portfolio` – сущности.
- `RiskManager`: проверяет лимиты, размер позиции, максимальную просадку перед отправкой ордера.

**Application Layer**:
- `StrategyRunner`: получает поток свечей из `MarketDataProvider`, прогоняет через активные стратегии, генерирует сигналы.
- `OrderExecutor`: превращает сигнал в ордер, отправляет через `ExchangeAdapter`, обновляет состояние.
- `StateManager`: хранит текущие позиции, баланс (в памяти + периодически синхронизирует в БД/Redis).

**Infrastructure Layer**:
- `ExchangeAdapter` (интерфейс): методы `fetch_ohlcv()`, `create_order()`, `get_balance()`. Реализации:
	- `BinanceTestnetAdapter(ccxt)`
	- `BybitTestnetAdapter(ccxt)`  
		Таким образом, добавить новую биржу – написать новый адаптер.
		
- `MarketDataProvider`: использует WebSocket бирж для получения -свечей (через `ccxt pro`) и передает их в движок через внутренний event loop.
- `EventBus`: внутри движка использует `asyncio.Queue` или Redis Pub/Sub для публикации событий (новая сделка, изменение баланса) во внешний мир (Backend API).
- `Persistence`: репозиторий для сохранения сделок в БД (через SQLAlchemy).

## Архитектура Backtest Engine 

Для проверки стратегий на истории мы не используем реальный движок с биржами, а создаем **BacktestEngine**, который заимствует ту же `Strategy` и `RiskManager`, но использует мок-адаптеры:

- `HistoricalDataProvider` читает CSV/Parquet файлы с историческими свечами (можно загрузить с Binance public data).
    
- `PaperExchangeAdapter` симулирует исполнение ордеров с учетом комиссий и проскальзывания.
    
- Результаты (сделки, equity-кривая, метрики) сохраняются в БД и возвращаются в UI для отображения.

## Архитектура Data&State Layer

**PostgreSQL**:

- Таблицы: strategies (конфигурации), trades, orders, backtest_results, market_data_cache (опционально).
- Управляется через Alembic миграции.

**Redis**:

- Кэш текущих балансов и позиций (обновляется движком каждые N секунд).
- Pub/Sub канал engine_events для передачи событий Backend API -> WebSocket.
- Хранилище состояния для бэктестов (опционально).

### 🧩 Общая схема распределения ответственности

В системе три ключевых процесса (или группы потоков):

1. **Backend API (FastAPI)** — обслуживает клиентское веб-приложение, управляет конфигурациями, предоставляет исторические данные и ретранслирует события от движка в реальном времени.
2. **Trading Engine (Python-процесс)** — содержит бизнес-логику стратегий, взаимодействует с биржами, генерирует сигналы и ордера, управляет состоянием.
3. **Базы данных и кэш** — PostgreSQL (долговременное хранение) и Redis (оперативный кэш, Pub/Sub, хранение состояния).

Связь между Backend API и Trading Engine — **асинхронная**, через Redis Pub/Sub и общую базу данных (PostgreSQL). Такая архитектура позволяет им работать независимо и переживать временные отключения друг друга.

---

### 🌐 Backend API (FastAPI)

**Назначение:** REST-сервер, который является единственной точкой входа для React-фронтенда. Он не содержит торговой логики, но управляет её конфигурацией и предоставляет результаты.

#### Модули и их данные

1. **Контроллеры стратегий (`/api/strategies`)**
   - **Входящие данные (от фронтенда):** JSON с полной конфигурацией стратегии (все параметры из формы создания: биржа, рынок, пара, тип стратегии, параметры, риск-менеджмент, название).
   - **Исходящие данные:** Список сохранённых стратегий с их статусами, метриками (доходность, P&L).
   - **Что делает:** Сохраняет конфигурацию в таблицу `strategies` PostgreSQL. При запуске/остановке/изменении статуса обновляет запись и отправляет команду движку через Redis Pub/Sub (канал `engine_commands`). Также читает текущий статус и метрики (доходность, количество сделок) из PostgreSQL (обновляется движком).

2. **Контроллер аккаунта (`/api/account`)**
   - **Исходящие данные:** Общий баланс в USDT, доступная маржа (для фьючерсов), список активов с их балансами.
   - **Что делает:** Не идёт напрямую на биржу. Запрашивает данные из Redis (ключ `account:{exchange}:{testnet}`), куда движок периодически (раз в 5 секунд) сохраняет снимок баланса, полученный через `ccxt` с тестовой биржи. Если в Redis пусто, может дёрнуть движок напрямую через временный HTTP-эндпоинт, но основной путь — через кэш.

3. **Контроллер сделок (`/api/trades`)**
   - **Исходящие данные:** Список сделок с фильтрацией (по дате, паре, стратегии) и пагинацией. Агрегированные показатели (общее количество, объём, комиссии).
   - **Что делает:** Читает таблицу `trades` в PostgreSQL, куда движок пишет каждую совершённую сделку. Может кэшировать агрегации в Redis для ускорения. Поддерживает экспорт в CSV.

4. **WebSocket-менеджер (`/ws/updates`)**
   - **Входящие данные:** Подписка на события от фронтенда (постоянное соединение).
   - **Исходящие данные:** JSON-сообщения о новых сделках, изменении баланса, изменении статуса движка, новых логах.
   - **Что делает:** Подписывается на Redis Pub/Sub канал `engine_events`, получает события от движка и пересылает их всем подключённым WebSocket-клиентам. Также может принимать команды от фронтенда и транслировать их движку через Redis.

5. **Контроллер бэктестинга (`/api/backtest`)**
   - **Входящие данные:** Параметры бэктеста (аналогично стратегии + диапазон дат, начальный депозит).
   - **Исходящие данные:** Результаты бэктеста (метрики, график эквити, список сделок).
   - **Что делает:** Может запускать бэктест двумя способами:
     - *Прямой вызов модуля бэктестера внутри API-процесса* (проще для курсовой).
     - *Асинхронный запуск через движок* (отправить задачу, потом опрашивать результат).
     Возвращает сохранённый в БД результат бэктеста (таблица `backtest_results`).

6. **Логирование**
   - **Исходящие данные:** Поток логов движка в реальном времени (для страницы «Логи»).
   - **Что делает:** Подписывается на Redis Pub/Sub канал `logs` (куда движок публикует структурированные логи) и рассылает их через WebSocket или предоставляет REST-эндпоинт для истории логов (хранятся в файлах или БД).

### 🧠 Trading Engine (ядро)

**Назначение:** Автономный процесс (или несколько), который непрерывно работает с биржами, исполняет стратегии и генерирует ордера. Он полностью изолирован от фронтенда и общается через БД и Redis.

#### Слои и источники данных

**1. Инфраструктурный слой — Exchange Adapters**
- **Получает данные от биржи:**
  - **Котировки и свечи:** Подключается к WebSocket-потокам Binance/Bybit Testnet через `ccxt.pro` и получает реальное время: `OHLCV` для заданных пар и таймфреймов. Адаптер `BinanceTestnetAdapter` знает специфичный URL testnet.
  - **Баланс и позиции:** Периодически (REST) запрашивает `fetch_balance()` и `fetch_positions()` и помещает в Redis (`account:{exchange}`).
  - **Исполнение ордеров:** Принимает команду `create_order()` от слоя Application, отправляет её через `ccxt` и возвращает результат (ID ордера, статус, исполненный объём).
- **Что публикует:** События о новых свечах, изменении баланса внутрь движка, а также ошибки подключения.

**2. Инфраструктурный слой — Market Data Provider**
- **Внутренний маршрутизатор свечей:** Получает тики/свечи от адаптеров и рассылает их всем активным стратегиям через `asyncio.Queue` на каждый `StrategyRunner`. Это гарантирует, что все стратегии, работающие на одной паре и таймфрейме, получают одну и ту же свечу одновременно.

**3. Domain Layer (стратегии) — реализация `Strategy`**
- **Получает:** Объект `Candle` (open, high, low, close, volume, timestamp) от Market Data Provider.
- **Обрабатывает:** Рассчитывает индикаторы (SMA, RSI, Grid-уровни, DCA-расписание) и хранит историю свечей в памяти (например, последние 200 свечей для расчёта SMA 200).
- **Генерирует:** Объект `Signal` (название стратегии, пара, тип BUY/SELL, объём, цена, метаданные) или `None`, если сигнала нет.

**4. Application Layer — Strategy Runner**
- **Получает:** Список активных стратегий (из БД или конфигурации, которую движок загружает при старте и обновляет по командам Redis).
- **Что делает:** Для каждой активной стратегии создаёт свой цикл обработки свечей. При появлении сигнала передаёт его `RiskManager`, затем `OrderExecutor`.
- **State Manager:** Хранит в памяти текущие позиции и баланс (синхронизирует с биржей), обновляет их после исполнения сделок. Периодически сбрасывает состояние в Redis.

**5. Application Layer — Risk Manager**
- **Получает:** Сигнал от стратегии и текущее состояние позиций, баланса, глобальных лимитов.
- **Проверяет:**
  - Достаточно ли баланса для открытия позиции.
  - Не превышен ли максимальный размер позиции (% от капитала).
  - Не нарушает ли сигнал стоп-лосс / тейк-профит уже имеющихся позиций.
  - Корректен ли объём (лотность, минимум биржи).
- **Возвращает:** Либо разрешённый, возможно скорректированный сигнал, либо отказ с указанием причины (пишется в лог).

**6. Application Layer — Order Executor**
- **Получает:** Подтверждённый сигнал от Risk Manager.
- **Что делает:** Преобразует сигнал в параметры ордера (тип: лимитный/рыночный, цена, объём). Вызывает метод адаптера биржи. По факту исполнения (или частичного исполнения) создаёт запись в таблице `trades` PostgreSQL, публикует событие `new_trade` в Redis (канал `engine_events`), обновляет состояние позиций.

**7. Мониторинг и управление жизненным циклом**
- **Слушает Redis Pub/Sub канал `engine_commands`:** Команды `start_strategy`, `stop_strategy`, `pause_strategy`, `update_config`. При получении команды движок загружает/выгружает стратегию из памяти, обновляет конфиг в БД.
- **Публикует свой статус** (Online, количество стратегий) в Redis, чтобы API мог отобразить на Dashboard.

### 🔄 Потоки данных (пример: проход от свечи до сделки)

1. **Биржа** → WebSocket → **Exchange Adapter** → свеча `BTC/USDT 15m`.
2. **Market Data Provider** → кладёт свечу в очередь стратегии `SMA Cross`.
3. **Strategy Runner** подаёт свечу в метод `on_candle()` стратегии `SMA Cross`.
4. Стратегия рассчитывает SMA50 и SMA200, сравнивает, генерирует **Signal BUY**.
5. **Risk Manager** проверяет баланс, лимиты, подтверждает сигнал.
6. **Order Executor** вызывает `BinanceTestnetAdapter.create_order(...)`, получает Order ID.
7. После исполнения ордера пишет сделку в **PostgreSQL** (таблица `trades`).
8. Публикует событие `new_trade` в **Redis** (канал `engine_events`).
9. **Backend API** слушает этот канал, забирает событие и через WebSocket отправляет клиенту.
10. **React-фронтенд** получает сообщение и добавляет запись в таблицу последних сделок.

Также параллельно движок раз в 5 секунд обновляет баланс в Redis, а API при запросе `/api/account` читает оттуда.

### 📊 Кто какие данные хранит

| Хранилище | Данные | Кто пишет | Кто читает |
|:---|:---|:---|:---|
| **PostgreSQL** `strategies` | Конфигурации стратегий, статус, время создания | API | API, движок (при запуске) |
| **PostgreSQL** `trades` | Все совершённые сделки (время, пара, тип, цена, объём, комиссия, стратегия) | Движок (после исполнения) | API (для отображения и бэктестов) |
| **PostgreSQL** `backtest_results` | Результаты бэктестов (метрики, графики, сделки) | API (или бэктестер) | API |
| **Redis** `account:{exchange}` | Баланс и позиции в JSON | Движок (каждые 5 сек) | API |
| **Redis** `engine_status` | Статус движка (online, uptime, активные стратегии) | Движок (heartbeat) | API |
| **Redis Pub/Sub** `engine_events` | События: новая сделка, изменение баланса, лог | Движок | API (для WebSocket) |
| **Redis Pub/Sub** `engine_commands` | Команды запуска/остановки/изменения стратегий | API | Движок |

### 💡 Рекомендация для курсовой

При описании этой архитектуры в работе можно сделать акцент на том, что:

- **Разделение на API и Engine** повышает надёжность: сбой веб-сервера не останавливает торговлю.
- **Использование Redis Pub/Sub** позволяет строить событийно-ориентированную систему без прямых HTTP-зависимостей между сервисами.
- **Адаптеры бирж** реализуют полиморфизм — легко добавить третью биржу, не меняя ядро.
- **Единый код стратегий** используется и в live-торговле, и в бэктестере за счёт того, что источник данных и исполнение ордеров заменяются мок-объектами (PaperExchangeAdapter).
