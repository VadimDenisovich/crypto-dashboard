
# План курсовой 

## Введение 

- Актуальность автоматизации торговли на криптовалютных биржах.
- Проблема: высокий порог входа для использования готовых ботов, отсутствие прозрачности и гибкости.
- Цель: разработать собственную событийно-ориентированную систему алгоритмической торговли с веб-интерфейсом, способную работать в тестовых средах (песочницах) бирж Binance и Bybit.
- Задачи:
	1. Проанализировать предметную область, принципы спотовой торговли и существующие алгоритмические стратегии.
	2. Спроектировать архитектуру системы, объединяющую торговое ядро, сервер API и клиентский веб-интерфейс.
	3. Реализовать торговый движок с подключаемыми адаптерами бирж и набором базовых стратегий.
	4. Разработать веб-панель управления для конфигурации стратегий, мониторинга позиций и визуализации сделок.
	5. Провести тестирование системы в песочницах Binance Testnet и Bybit Testnet, а также выполнить бэктестинг на исторических данных.
- Объект исследования: процесс алгоритмической торговли криптоактивами.
- Предмет исследования: архитектура, методы и инструменты построения автоматизированной торговой системы с веб-управлением.

## Глава 1. Анализ предметной области и постановка задачи 

- 1.1. Основы спотовой торговли на криптобиржах: ордера, стакан, комиссии, ликвидность.
- 1.2. Обзор алгоритмических стратегий для спотового рынка:
    - Трендовые стратегии (SMA Cross, MACD).
    - Стратегии возврата к среднему (RSI, полосы Боллинджера).
    - Выбор базовых стратегий для реализации (обоснование простоты и наглядности).
- 1.3. Обзор инструментов для создания торговых систем:
    - Готовые open-source боты (Freqtrade, Hummingbot) — их достоинства и ограничения.
    - Языки и библиотеки: Python с `ccxt`, `FastAPI`, `pandas`; React для пользовательского интерфейса.
    - Тестовые среды бирж (Binance Testnet, Bybit Testnet) как безопасный полигон для отладки.
- 1.4. Анализ требований к разрабатываемой системе:    
    - Функциональные требования: запуск/остановка стратегий, работа с несколькими биржами и парами, real-time мониторинг, бэктестинг.
    - Нефункциональные требования: отказоустойчивость, расширяемость, контейнеризация, удобный UI.
- Выводы по главе и уточнённая постановка задачи.

#### Глава 2. Проектирование архитектуры системы

- 2.1. Общая архитектура: событийно-ориентированный подход, разделение на независимые компоненты.
- 2.2. Компонентная схема и взаимодействие:
    - **Trading Engine** — ядро, реализующее стратегии и управляющее ордерами.
    - **Backend API** (FastAPI) — REST и WebSocket-интерфейс для фронтенда.
    - **Frontend** (React SPA) — дашборд и панель управления.
    - **Слой хранения** — PostgreSQL (конфигурации, история сделок) и Redis (кэш состояния, Pub/Sub).
- 2.3. Детальное проектирование Trading Engine:
    - Паттерн «Чистая архитектура»: Domain (сущности Strategy, Signal, Order), Application (StrategyRunner, OrderExecutor, RiskManager), Infrastructure (ExchangeAdapter, MarketDataProvider).
    - Полиморфизм адаптеров бирж: `BinanceTestnetAdapter`, `BybitTestnetAdapter`.
    - Поток событий: получение свечей, расчёт индикаторов, генерация сигналов, проверка рисков, исполнение ордера, публикация результата.
- 2.4. Проектирование Backend API и взаимодействия в реальном времени:
    - REST-методы для CRUD стратегий и получения истории.
    - WebSocket-канал `/ws/updates`: подписка на события движка через Redis Pub/Sub, push-уведомления клиенту.
- 2.5. Проектирование Frontend:
    - Структура страниц: Dashboard, Управление стратегиями, История сделок, Бэктестинг.
    - Компоненты визуализации: графики курса и индикаторов, таблица ордеров, кривая доходности.
- 2.6. Модуль бэктестинга:
    - Использование общего кода стратегий с подменой источника данных на исторический и симуляцией исполнения (PaperExchangeAdapter).
    - Хранение и отображение метрик (Profit Factor, Max Drawdown, Sharpe Ratio).
- 2.7. Выбор технологического стека: Python 3.11, FastAPI, SQLAlchemy, Alembic, `ccxt` + `ccxt.pro`, React 18, TypeScript, Docker Compose, PostgreSQL, Redis.

#### Глава 3. Реализация системы

- 3.1. Настройка окружения и контейнеризация: описание Docker Compose файлов и переменных окружения.
- 3.2. Реализация инфраструктурного слоя:
    - Адаптеры бирж: методы `fetch_ohlcv`, `create_order` для Binance и Bybit Testnet.
    - Провайдер рыночных данных через WebSocket (`ccxt.pro`), восстановление соединения.
    - Репозитории для работы с PostgreSQL (модели стратегий, ордеров).
    - Клиент Redis для кэширования и Pub/Sub.
- 3.3. Реализация доменного слоя и Application-логики:
    - Классы стратегий (SMA Cross, RSI) — код и пояснение логики генерации сигналов.
    - RiskManager: контроль минимального лота, максимальной позиции, стоп-лосса.
    - StrategyRunner: асинхронный цикл обработки свечей.
    - OrderExecutor: преобразование сигнала в лимитный/рыночный ордер с обработкой ответа биржи.
- 3.4. Реализация Backend API:
    - Маршруты `/api/strategies`, `/api/account`, `/api/trades`, `/api/backtest`.
    - WebSocket endpoint и трансляция событий из Redis в клиентские подключения.
    - Генерация документации через Swagger UI.
- 3.5. Реализация Frontend:
    - Структура React-приложения (компоненты, хуки, контекст).
    - Real-time обновления через WebSocket (кастомный хук `useWebSocket`).
    - Визуализация графиков (свечной график, наложение индикаторов) с помощью библиотеки Lightweight Charts.
- 3.6. Реализация модуля бэктестинга:
    - Загрузка исторических данных из файлов (CSV/Parquet) или через REST API бирж.
    - Имитация исполнения ордеров с учётом комиссий и проскальзывания.
    - Расчёт метрик и формирование отчёта.

#### Глава 4. Тестирование и анализ результатов

- 4.1. Сценарии и параметры тестирования:
    - Пары: BTC/USDT, ETH/USDT.
    - Таймфреймы: 5m, 1h.
    - Стратегии: SMA(50,200), RSI(14) с уровнями 30/70.
- 4.2. Тестирование в песочнице Binance Testnet:
    - Запуск стратегий на протяжении заданного периода, логирование сделок.
    - Верификация поведения: соответствие сигналов индикаторам, корректность исполнения ордеров.
- 4.3. Тестирование в песочнице Bybit Testnet:
    - Аналогичный запуск для демонстрации работы адаптера другой биржи.
    - Сравнение показателей комиссий и скорости исполнения (обзорно).
- 4.4. Бэктестинг стратегий на исторических данных:
    - Результаты моделирования за 6–12 месяцев.
    - Ключевые метрики: Profit Factor, Sharpe Ratio, Maximum Drawdown, общая доходность, процент прибыльных сделок.
    - Сравнение эффективности стратегий в разных рыночных условиях.
- 4.5. Оценка пользовательского интерфейса: удобство мониторинга и настройки, юзабилити-выводы.

#### Заключение

- Итоги выполнения поставленных задач.
- Основные результаты: разработана полнофункциональная система с микросервисной архитектурой, подтверждена её работоспособность на тестовых средах двух бирж.
- Достоинства подхода: расширяемость за счёт адаптеров, единый код стратегий для live и бэктеста.

### 📈 Как это подать комиссии: акценты

1. **Clean Architecture + DDD** – покажи диаграмму слоев, объясни, что бизнес-логика не зависит от биржи и UI.
2. **Event-driven** – асинхронная связь через Redis Pub/Sub, WebSocket для real-time обновлений.
3. **Полиморфизм бирж** – благодаря паттерну «Адаптер» легко добавлять новые биржи.
4. **Единый код стратегий** – одни и те же классы используются и в live, и в бэктесте.
5. **Контейнеризация** – Docker Compose для быстрого развертывания.
