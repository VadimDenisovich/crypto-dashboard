## 1. Что мы вообще сделали

Мы построили **ядро** торгового бота. Не сам бот — а его «мозг».
Бот пока ещё ничего не торгует, потому что мы не подключили его к
бирже и базе данных. Но мы заложили все правила, по которым он
будет работать, когда мы их подключим.

Аналогия: представь, что ты строишь дом.
- **Сейчас мы залили фундамент и поставили несущие стены** — это и есть «ядро».
- Электрика, отопление, мебель (биржа Binance, база данных PostgreSQL,
  Redis, фронтенд) — это уже следующие шаги.
- Главное преимущество такого подхода: фундамент **не зависит** от того,
  какую плитку ты потом положишь. Сменишь Binance на Bybit — фундамент
  не трогаешь.

## 2. Зачем нужны «слои» (Clean Architecture)

Код разделён на три кольца, как луковица:

```
┌─────────────────────────────────────────────┐
│   INFRASTRUCTURE (внешний мир — потом)      │
│  ┌─────────────────────────────────────┐    │
│  │  APPLICATION (бизнес-процессы)      │    │
│  │  ┌───────────────────────────────┐  │    │
│  │  │  DOMAIN (правила и сущности)  │  │    │
│  │  └───────────────────────────────┘  │    │
│  └─────────────────────────────────────┘    │
└─────────────────────────────────────────────┘
```

**Главное правило:** внутренние слои **ничего не знают** о внешних.
- `domain/` — самый внутренний. Тут только чистая Python-логика и описания.
- `application/` — может пользоваться `domain/`, но не знает про CCXT/Redis/БД.
- `infrastructure/` (мы её ещё не написали) — будет реализовывать конкретные
  биржи, БД, Redis. Она «снаружи» подключается к `application` через интерфейсы.

**Зачем это надо?** Чтобы тестировать без биржи и без интернета. Чтобы
завтра поменять Binance на OKX за один день. Чтобы один и тот же код
стратегии работал и в реальной торговле, и в бэктесте на истории.

## 4. Domain — что внутри каждого файла

### `src/domain/enums.py` — «справочники»

Это просто **списки разрешённых значений**, чтобы не писать строки руками.

- `Side` — направление: `BUY` (купить) или `SELL` (продать).
- `OrderType` — тип ордера: `MARKET` (по рынку, цена не задана) или
  `LIMIT` (с явной ценой).
- `OrderStatus` — статус ордера: `NEW`, `OPEN`, `PARTIAL`,
  `FILLED`, `CANCELED`, `REJECTED`, `EXPIRED`.
- `TimeFrame` — таймфрейм свечи: `M1`, `M5`, `M15`, `H1`, `H4`, `D1`.

Все эти штуки сделаны как `StrEnum` — это значит, что `Side.BUY == "buy"`.
Удобно сравнивать и сериализовать в JSON.

**Зачем это вообще?** Чтобы нельзя было нечаянно написать `side="buyy"`
с опечаткой. Линтер сразу подсветит ошибку.

### `src/domain/models.py` — «коробочки с данными»

Тут лежат структуры данных, которыми гоняет весь движок. Каждая —
это `@dataclass(frozen=True, slots=True)` с `Decimal` для денег.
Сейчас расшифрую, что это значит.

**`Candle`** — одна свеча на графике:
```python
Candle(symbol, timeframe, timestamp, open, high, low, close, volume)
```
Биржа присылает их каждые N минут. На основе свечей стратегия принимает
решение «купить или продать».

**`Signal`** — сигнал от стратегии:
```python
Signal(strategy_name, symbol, side, size, price, metadata)
```
Когда стратегия решила «надо купить» — она возвращает `Signal`. Это ещё
не ордер на бирже, это просто намерение.

**`Order`** — реальный ордер на бирже:
```python
Order(order_id, symbol, side, type, size, status, filled_size, price, fee)
```
Это уже то, что отправлено на биржу и имеет ID.

**`Position`** — открытая позиция (купили и держим):
```python
Position(symbol, side, entry_price, size, current_pnl)
```

**`Balance`** — состояние счёта по одной валюте:
```python
Balance(currency, free, used, total)
```
`free` — свободно, `used` — заморожено в открытых ордерах, `total` — сумма.

#### Расшифровка магических слов

**`@dataclass`** — декоратор Python, который автоматически делает
конструктор, метод сравнения и вывод. То есть вместо `__init__` руками
ты просто описываешь поля, и Python всё сгенерирует.

**`frozen=True`** — после создания объект **нельзя изменить**. Если
попытаешься `candle.close = 999`, Python кинет ошибку. Это важно для
надёжности: данные о свече не должны меняться задним числом.

**`slots=True`** — оптимизация памяти и скорости. Без `slots` каждый
объект имеет внутри словарь `__dict__`, что сжирает много памяти. Со
`slots` — нет словаря, поля жёстко зафиксированы. Для миллионов свечей
в бэктесте это критично.

**`Decimal` (а не `float`)** — это **главное правило для денег**.
`float` — это «приблизительное число» в Python: `0.1 + 0.2 == 0.30000000000000004`.
Если так считать деньги, у тебя через год накопится ошибка в копейки,
а потом и в рубли. `Decimal` — точное число, которое хранит цифры как
есть. **В коде всегда `Decimal("100.50")` через строку**, а не `Decimal(100.50)`.

**`__post_init__`** — это метод, который Python автоматически
вызывает сразу после создания dataclass'а. Мы используем его, чтобы
**проверить инварианты**. Например: у свечи `high` не может быть меньше
`low` — это бессмыслица. Или у баланса `free + used` обязано совпадать
с `total`. Если правило нарушено — поднимаем `ValueError`. Объект просто
не создастся.

### `src/domain/exceptions.py` — «свои собственные ошибки»

В Python есть стандартные ошибки (`ValueError`, `RuntimeError`, ...).
Мы добавили свои, чтобы было понятно, **что именно** пошло не так:

- `DomainError` — корень всей нашей иерархии ошибок.
- `RiskRejectedError` — RiskManager отказал в сделке (с полем `reason`).
- `InsufficientBalanceError` — наследник `RiskRejectedError`, специально
  для случая «не хватает денег».
- `LotSizeError` — наследник `RiskRejectedError`, для случая «слишком
  маленький лот».
- `InvalidSignalError` — стратегия выдала кривой сигнал.
- `OrderExecutionError` — биржа не приняла ордер (сеть упала, например).

**Зачем это надо?** Чтобы в `try/except` можно было ловить **именно тот
тип ошибки**, который нам нужен, а не «вообще все ошибки подряд».
Например, в `StrategyRunner` мы ловим `RiskRejectedError` и
`OrderExecutionError`, чтобы цикл стратегии не падал из-за одной плохой
свечи. А вот баги в коде (RuntimeError) мы **не глотаем** — пусть летят
наверх, потому что это нужно чинить.

### `src/domain/interfaces.py` — «контракты» (ABC)

Это **самая важная** часть Clean Architecture. Тут мы **описываем**, что
должна уметь биржа, стратегия, источник данных и шина событий — но **не
реализуем** ни одну из них. Реализации придут позже в `infrastructure/`.

**`ABC` (Abstract Base Class)** — это «абстрактный класс». Его нельзя
создать напрямую: `Strategy()` — будет ошибка. От него можно только
**унаследоваться** и реализовать его методы.

**`@abstractmethod`** — пометка «этот метод обязан быть реализован в
наследнике».

В файле описаны 4 контракта:

**`ExchangeAdapter`** — что должна уметь биржа:
- `fetch_ohlcv(...)` — скачать историю свечей.
- `create_order(...)` — отправить ордер.
- `get_balance()` — получить баланс.

Когда мы потом напишем `BinanceAdapter`, он будет наследоваться от
`ExchangeAdapter` и реализовывать эти 3 метода. То же — `BybitAdapter`,
`PaperExchangeAdapter` (для бэктеста).

**`MarketDataProvider`** — источник свечей в реальном времени:
- `subscribe(symbol, timeframe)` возвращает асинхронный генератор свечей.

Реальная реализация будет подключаться к WebSocket биржи через `ccxt.pro`.
Для бэктеста — будет читать CSV-файл.

**`Strategy`** — что должна уметь стратегия:
- `on_start()` — однократная подготовка (опционально).
- `on_candle(candle) -> Signal | None` — получает свечу, возвращает
  сигнал или `None` если ничего делать не надо.

Когда мы напишем стратегию `SmaCross`, она унаследуется от `Strategy` и
реализует `on_candle`.

**`EventBus`** — шина событий:
- `publish(channel, payload)` — опубликовать событие в канал.

Реальная реализация будет публиковать в Redis Pub/Sub. Для тестов —
заменяется моком.

#### Зачем все эти ABC?

Чтобы наш Application слой работал с **абстракцией**, а не с конкретной
библиотекой. Application'у плевать, Binance это или Bybit. Он работает
через интерфейс. **Это и есть «инверсия зависимостей»** — внешний код
зависит от внутреннего, а не наоборот.

## 5. Application — бизнес-процессы

### `src/application/events.py` — «названия каналов»

Просто 4 строки-константы:
- `NEW_TRADE = "engine.new_trade"` — публикуется при исполнении ордера.
- `BALANCE_UPDATE = "engine.balance_update"` — при изменении баланса.
- `ENGINE_STATUS = "engine.status"` — heartbeat движка.
- `STRATEGY_ERROR = "engine.strategy_error"` — что-то пошло не так в стратегии.

В будущем эти строки будут именами **Redis Pub/Sub каналов**, на
которые подпишется Backend API и через WebSocket отдаст фронту.

### `src/application/risk_manager.py` — «вышибала»

`RiskManager` — это **проверяющий перед сделкой**. Его задача —
не дать боту слить депозит из-за глупой ошибки в стратегии.

Создаётся так:
```python
RiskManager(adapter, max_position_pct=0.1, min_lot=0.001)
```
- `adapter` — биржа (через интерфейс), нужна чтобы дёрнуть актуальный баланс.
- `max_position_pct=0.1` — не давать сделке больше, чем 10% от капитала.
- `min_lot=0.001` — отказывать в сделках меньше 0.001 BTC (биржа всё
  равно отклонит).

Метод `await risk.check(signal)` делает 4 проверки:
1. **Размер сделки** — не меньше `min_lot`. Если меньше → `LotSizeError`.
2. **Цена обязана быть** — `RiskManager` не работает с маркет-ордерами,
   потому что без цены нельзя посчитать notional (объём в долларах).
3. **Баланс**:
   - `BUY BTC/USDT` → проверяет, что `quote-балансе (USDT)` хватает на
     `size * price`.
   - `SELL BTC/USDT` → проверяет, что `base-балансе (BTC)` хватает на
     `size`.
4. **Лимит позиции**: `notional` не больше `max_position_pct *
   total_equity`. Если больше → `RiskRejectedError`.

Если все 4 проверки прошли — возвращает тот же сигнал. Если нет —
бросает соответствующее исключение.

### `src/application/order_executor.py` — «посыльный»

`OrderExecutor` берёт **одобренный** сигнал и отправляет на биржу.

```python
OrderExecutor(adapter, event_bus)
await executor.execute(signal)
```

Что делает:
1. Если у сигнала есть `price` → ордер `LIMIT`. Если нет → `MARKET`.
2. Зовёт `adapter.create_order(...)` с правильными параметрами.
3. Если адаптер кинул любое исключение — оборачивает в
   `OrderExecutionError` (с цепочкой через `__cause__`, чтобы не
   потерять оригинал).
4. На успехе — публикует событие `NEW_TRADE` в `event_bus` со всеми
   деталями ордера. Это событие потом улетит в Redis и далее на фронт.

### `src/application/strategy_runner.py` — «дирижёр»

Это **главный цикл движка**. По одному `StrategyRunner` на каждую
активную стратегию.

```python
runner = StrategyRunner(strategy, market_data, risk, executor, event_bus)
await runner.run()
```

Что происходит внутри `run()`:
1. Зовёт `strategy.on_start()` — однократная подготовка (загрузить
   историю, например).
2. Запускает бесконечный async-цикл:
   ```python
   async for candle in market_data.subscribe(symbol, timeframe):
       signal = strategy.on_candle(candle)
       if signal is None:
           continue
       try:
           approved = await risk.check(signal)
           await executor.execute(approved)
       except RiskRejectedError:
           # риск отказал — публикуем STRATEGY_ERROR, цикл продолжается
       except OrderExecutionError:
           # биржа не приняла — то же самое, цикл продолжается
   ```

**Главная фишка**: если на одной свече что-то пошло не так — цикл
**не падает**. Стратегия должна работать днями. Один кривой сигнал
не должен останавливать весь движок.

**Но** если случилось что-то реально странное (например, баг в коде —
`KeyError`, `ZeroDivisionError`) — мы **не глотаем** это исключение, а
пробрасываем наверх. Потому что баги надо чинить, а не маскировать.

## 6. Что такое «асинхронность» и зачем она тут

Python умеет два режима работы:
- **Синхронный** — код выполняется строка за строкой, и пока
  `time.sleep(1)` спит — программа стоит.
- **Асинхронный** (`async/await`) — пока одна задача ждёт ответа от
  биржи, другая может работать.

В торговом боте это **обязательно**, потому что:
- Свечи приходят по WebSocket — это сетевая операция.
- Запрос баланса — сетевая операция.
- Отправка ордера — сетевая операция.

Если делать синхронно, пока ждёшь ответа на ордер по BTC — упустишь свечи
по ETH. Поэтому весь интерфейс `ExchangeAdapter` — `async`, и
`StrategyRunner` — это `async def run()`.

Маркер: метод с `async def` — асинхронный, его надо вызывать через
`await`. Если в `interfaces.py` метод объявлен как `async def` — все его
реализации тоже обязаны быть `async def`.

## 7. Тесты — зачем и как

В папке `tests/` лежат **33 теста**, которые проверяют, что наш код
работает правильно **без интернета и без биржи**. Это критично, потому
что без тестов мы бы узнавали о багах только в продакшене на реальных
деньгах.

### Как тестируется код, который зависит от биржи?

Через **моки** (mock — «подделка»). Мы создаём фейковый адаптер,
который не лезет в интернет, а возвращает заранее подготовленные данные.

```python
adapter = AsyncMock(spec=ExchangeAdapter)
adapter.get_balance.return_value = {"USDT": Balance(...)}
```

`AsyncMock` — это «фейк, который умеет в async». Когда `RiskManager` зовёт
`await adapter.get_balance()`, он на самом деле зовёт мок, который
вернёт нашу подготовленную карту балансов.

`spec=ExchangeAdapter` — говорит моку «притворяйся, что ты
`ExchangeAdapter`, у тебя должны быть только его методы». Если в коде
случайно вызовут `adapter.несуществующий_метод()` — мок ругнётся.

### Что именно проверяем

- **`test_models.py`** (13 тестов) — что инварианты dataclass'ов
  работают: нельзя создать свечу с `high < low`, баланс с
  неправильной суммой, лимит-ордер без цены и т.д.
- **`test_risk_manager.py`** (12 тестов) — happy paths и **каждая ветка
  отказа**: маленький лот, нет баланса, превышен лимит позиции, кривой
  символ, неизвестная валюта.
- **`test_order_executor.py`** (3 теста) — успешный лимит-ордер,
  автоопределение market при price=None, обработка падения адаптера.
- **`test_strategy_runner.py`** (5 тестов) — поток сигнала, отсутствие
  сигнала, обработка `RiskRejectedError`, `OrderExecutionError`, и
  пропуск неизвестных исключений наверх.

Запуск:
```bash
.venv/bin/pytest tests/ -v
```

Должно показать `33 passed`.

## 8. Дополнительные проверки

### `mypy` — статическая проверка типов

Python — динамический язык, ошибки типов он находит только во время
запуска. `mypy` — это инструмент, который читает наши type hints и
проверяет, что мы нигде не нарушили типы. У нас включён `strict` режим,
самый суровый.

```bash
.venv/bin/mypy src/
# → Success: no issues found in 10 source files
```

### Аудит зависимостей

В правилах из `CLAUDE.md` сказано: в `src/domain/` запрещены импорты
`ccxt`, `sqlalchemy`, `aioredis`, `fastapi`, `pydantic`. Мы это
проверили — там только Python stdlib (`enum`, `dataclasses`, `decimal`,
`datetime`, `typing`, `abc`, `collections`). В `src/application/` тоже
только stdlib + `domain/`. Никакой утечки инфраструктуры.

## 9. Поток данных (как это всё работает вместе)

Допустим, биржа Binance прислала свежую 5-минутную свечу BTC/USDT:

```
1. WebSocket Binance
        ↓
2. BinanceAdapter (когда напишем) — конвертирует JSON в Candle
        ↓
3. MarketDataProvider — кладёт Candle в async-итератор
        ↓
4. StrategyRunner ловит свечу из `async for`
        ↓
5. Strategy.on_candle(candle) — стратегия думает, считает индикаторы
        ↓
   возвращает Signal(BUY, BTC/USDT, 0.01, 100000) или None
        ↓
6. RiskManager.check(signal) — проверяет баланс, лимиты
        ↓
   возвращает signal или бросает RiskRejectedError
        ↓
7. OrderExecutor.execute(signal) — отправляет на биржу
        ↓
8. ExchangeAdapter.create_order(...) — реальный вызов API биржи
        ↓
9. Order возвращается, OrderExecutor публикует NEW_TRADE в EventBus
        ↓
10. EventBus → Redis Pub/Sub → Backend API → WebSocket → React
```

Всё, что мы сейчас сделали — **шаги 4-7 и 9** (плюс контракты для 1-3
и 8-10). Реализации шагов 1, 2, 3, 8, 10 — это infrastructure, она
будет дальше.

## 10. Что мы НЕ сделали (и это нормально)

  `src/infrastructure/persistence/` и `src/infrastructure/messaging/`.

- **Backend API на FastAPI**. Отдельный сервис.
- **Бэктест-движок**. Тоже отдельно, но **переиспользует** наши
  `Strategy`, `RiskManager`, `OrderExecutor` через `PaperExchangeAdapter`.
