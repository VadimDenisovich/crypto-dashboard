# Описание проекта
Асинхронный алгоритмический торговый движок + FastAPI бэкенд + React фронтенд. 
Движок (Trading Engine) и веб-сервер (Backend API) работают как независимые процессы и общаются исключительно через PostgreSQL и Redis Pub/Sub.

# 🚫 КРИТИЧЕСКИЕ АРХИТЕКТУРНЫЕ ПРАВИЛА (CLEAN ARCHITECTURE)
1. **Изоляция Domain слоя:** В папке `domain/` ЗАПРЕЩЕНЫ импорты внешних библиотек инфраструктуры (никаких `ccxt`, `sqlalchemy`, `aioredis`, `fastapi`). Здесь находятся только абстрактные классы (`abc.ABC`), dataclasses и базовая логика.
2. **Внедрение зависимостей (Dependency Injection):** Domain и Application слои получают реализации адаптеров (биржи, БД) через аргументы при инициализации. Жесткое кодирование (hardcode) зависимостей запрещено.
3. **Разделение процессов:** Backend API не выполняет торговую логику. Trading Engine не обрабатывает HTTP-запросы клиентов.

# ⚡ АСИНХРОННОСТЬ И ПРОИЗВОДИТЕЛЬНОСТЬ
1. **Никакого блокирующего I/O:** Весь код должен быть строго асинхронным. Запрещено использовать `time.sleep` (только `asyncio.sleep`), `requests` (только `aiohttp` или асинхронные клиенты) и синхронные драйверы баз данных.
2. **Управление задачами (Event Loop):** Для запуска параллельных стратегий использовать `asyncio.gather` или `asyncio.TaskGroup`.
3. **Graceful Shutdown:** Все фоновые задачи и соединения (WebSockets CCXT, сессии БД, Redis) должны корректно закрываться при остановке сервиса (перехват `SIGINT`/`SIGTERM`), чтобы избежать зависания ордеров и утечек памяти.

# 💾 БАЗА ДАННЫХ И СОСТОЯНИЕ (POSTGRESQL + REDIS)
1. **SQLAlchemy 2.0 Async:** Использовать только современный синтаксис SQLAlchemy 2.0 (например, `async_session.execute(select(Model))`, никаких устаревших `.query()`).
2. **Миграции:** Любые изменения в моделях БД должны сопровождаться генерацией миграций Alembic.
3. **Redis Pub/Sub для событий:** Все события (новая свеча, сделка, ошибка) передаются между сервисами через Redis Pub/Sub, а не через прямые HTTP-вызовы.
4. **Состояние в памяти + кэш:** `StateManager` внутри движка держит состояние в памяти, но каждые 5-10 секунд делает снимок в Redis, чтобы FastAPI мог быстро отдавать балансы без нагрузки на движок.

# 📈 ИНТЕГРАЦИЯ С БИРЖАМИ (CCXT.PRO)
1. **Обработка ошибок:** Вызовы к биржам (CCXT) обязаны быть обернуты в блоки `try/except`. Сетевые ошибки (Timeout, RateLimit) не должны "ронять" весь движок — их нужно логировать и использовать retry-механизмы с экспоненциальной задержкой.
2. **Нормализация данных:** Адаптеры бирж (`ExchangeAdapter`) обязаны приводить специфичные данные Binance/Bybit к нашим единым доменным моделям (`Candle`, `Order`, `Balance`).
3. **Точность (Precision):** При отправке ордеров всегда использовать инструменты CCXT для округления цены и объема до разрешенного биржей шага (tick size / lot size), иначе ордер будет отклонен.

# 🧪 ТЕСТИРОВАНИЕ (TDD)
1. **Mocking:** При написании тестов для стратегий и RiskManager внешние сервисы (база данных, биржа) ОБЯЗАТЕЛЬНО должны быть замоканы (`unittest.mock.MagicMock` / `AsyncMock`).
2. **Покрытие:** Бизнес-логика (особенно `RiskManager` и логика генерации сигналов `Strategy`) должна покрываться тестами через `pytest` ДО написания интеграции с реальными биржами.
